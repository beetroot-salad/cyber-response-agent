1	# Query template catalog — file format
2	
3	This directory is the gather subagent's accumulated memory of useful queries,
4	organized per system of record. A template captures what an experienced
5	analyst already knows about a data source: the right index, the right field
6	names, the joins or regex tricks that took someone an afternoon to figure
7	out the first time. Gather binds parameters and runs; it doesn't re-derive
8	the query under time pressure.
9	
10	## Layout
11	
12	```
13	queries/
14	  SCHEMA.md                    # this file
15	  {system}/                    # one dir per system of record
16	    {template-id}.md           # one file per template
17	```
18	
19	`{system}` is a system the gather subagent knows how to dispatch against
20	(e.g. `wazuh/`, `host-query/`). It doubles as the routing prefix for the
21	template id and as a coarse `ls`-time filter.
22	
23	`{template-id}` is kebab-case. Name it for **what the query measures**
24	(`auth-events`), not the axis you happen to filter on (`auth-events-by-host`)
25	and not why the defender asked for it (`check-if-bastion-was-pivoted`). A
26	single template can carry several optional filter knobs.

75	## Multi-query dispatches and `gather_raw/` naming
76	
77	A dispatch at sequence `position` typically runs one query and persists raw
78	to `{run_dir}/gather_raw/{position}.json`. When the lead requires more than
79	one query at the same position (composition leads, foreground+baseline,
80	multi-system comparisons), suffix each query's position with a single
81	lowercase letter so the run dir stays unambiguous:
82	
83	- `gather_raw/0.json` — single-query dispatch at position 0
84	- `gather_raw/0a.json`, `gather_raw/0b.json`, `gather_raw/0c.json` — three
85	  queries at position 0
86	- `gather_raw/1.json` — single-query dispatch at position 1
87	
88	System CLIs accept `--position <pos>` (in addition to `--run-dir`) and write
89	the file themselves; gather does not redirect stdout. The `lead_sequence`
90	projection groups files back under one `position` regardless of suffix.

92	## What is *not* a template
93	
94	Templates measure **primitives** — a single dataset, a single filter shape,
95	one CLI invocation. Cross-primitive correlations are not templates.
96	
97	When a lead asks for "X correlated with Y at time T" (e.g. *who was logged
98	in when /etc/passwd changed?*), the right move is: run the two primitives
99	that already exist, characterize the join in the gather summary. **Do not
100	mint a "bridge" template** — it bloats the catalog with one-offs that
101	won't be reused. See `defender/skills/gather/SKILL.md` §Composition leads.
102	
103	## Authoring a new template
104	
105	When the lead has no matching template:
106	
107	1. Pick a `{system}` based on which data source the query must hit.
108	2. Pick a kebab-case `{template-id}` describing what the query measures,
109	   not why the defender asked for it. Good: `auth-events`,
110	   `file-integrity-changes`. Bad: `check-bastion-pivot`,
111	   `auth-events-by-host` (the by-X axis is a parameter, not a separate
112	   template).
113	3. Write `## Goal` for keyword recall. Future-you will grep this body.
114	4. Run it, summarize for the defender, and record the id in
115	   `lead_sequence.yaml` like any other entry.
116	
117	Bias toward authoring a fresh id rather than wedging a near-match —
118	duplicates are cheaper to normalize later than mis-keyed cross-case joins
119	are to recover. But before authoring, check whether an existing template
120	already carries the **capability** you need with a different parameter

6	Wazuh is the org's primary SIEM. Authentication events, file integrity

9	## CLI
10	
11	The defender and gather dispatch Wazuh queries through the production
12	adapter at `soc-agent/scripts/tools/wazuh_cli.py`:
13	
14	```bash
15	python3 soc-agent/scripts/tools/wazuh_cli.py query \
16	  --query '<Lucene>' \
17	  --window 2h \
18	  --run-dir {run_dir}
19	```
20	
21	Common subcommands:
22	
23	- `query` — search the alerts index. `--query` is polymorphic:
24	  - **Lucene string** (`rule.id:5710 AND agent.name:web-03`) — the CLI
25	    wraps it in a bool with the time-range filter from `--start` /
26	    `--end` / `--window`. Best for "show me events" leads.
27	  - **JSON search body** (`'{"query": {...}, "aggs": {...}}'`) — the
28	    agent owns the entire body, including time filtering. Use this
29	    whenever the lead asks for counts, top-N, or distributions over a
30	    population that may exceed `--limit` — server-side aggs return true
31	    totals over the full match set, while the default Count Breakdown
32	    is a post-`--limit` sample.
33	  Other flags: `--limit` (default 500, max 10000; use `0` for
34	  count+aggs only), `--run-dir` (persists raw payload under
35	  `{run_dir}/gather_raw/` and wraps stdout in salted untrusted-data
36	  delimiters), `--raw` (rarely needed; default already embeds first 3
37	  events' `_source`).
38	- `health-check` — connectivity probe; the defender does not need to
39	  invoke this directly during a run.
40	
41	Lucene is OpenSearch query_string syntax: `rule.groups:sshd AND
42	data.srcip:10.0.0.5`. JSON bodies use the standard OpenSearch search
43	DSL — `query.bool.must` / `query.bool.filter`, `aggs.{name}.terms` /
44	`date_histogram` / `cardinality`, etc.
45	
46	Working query examples + field-level pitfalls live with the templates
47	under `defender/skills/gather/queries/wazuh/`. Reach for those when
48	authoring or grepping for a Wazuh measurement; this SKILL covers only
49	the system surface.

5	## Goal
6	
7	Retrieve recent fires of a specific Wazuh rule across the fleet over a
8	time window. Used to gauge whether the alert under investigation is a
9	one-off or part of a broader pattern (and where else it has fired).
10	
11	## What to characterize
12	
13	- Total fire count
14	- Hosts affected (count + top names)
15	- Source IPs (count + top sources)
16	- Timing — burst around the alert timestamp, or steady drumbeat?
17	
18	## Query
19	
20	For a small window where match_count is likely ≤ `--limit`, the Lucene
21	form is fine — the default Count Breakdown will reflect reality:
22	
23	```bash
24	python3 soc-agent/scripts/tools/wazuh_cli.py query \
25	  --query 'rule.id:${rule_id}' \
26	  --window ${window} \
27	  --run-dir ${run_dir}
28	```
29	
30	For wider windows or noisy rules where match_count likely exceeds
31	`--limit`, pass a JSON search body so the host / srcip / hour
32	distributions come back as server-side true totals instead of a
33	limit-capped sample:
34	
35	```bash
36	python3 soc-agent/scripts/tools/wazuh_cli.py query \
37	  --query '{
38	    "query": {
39	      "bool": {
40	        "must":   [{"query_string": {"query": "rule.id:${rule_id}"}}],
41	        "filter": [{"range": {"timestamp": {"gte": "${start}", "lte": "${end}"}}}]
42	      }
43	    },
44	    "aggs": {
45	      "by_agent":  {"terms": {"field": "agent.name",   "size": 10}},
46	      "by_srcip":  {"terms": {"field": "data.srcip",   "size": 10}},
47	      "by_user":   {"terms": {"field": "data.srcuser", "size": 10}},
48	      "by_hour":   {"date_histogram": {"field": "timestamp", "fixed_interval": "1h"}}
49	    }
50	  }' \
51	  --limit 5 \
52	  --run-dir ${run_dir}
53	```
54	
55	`--limit 5` keeps a handful of recent events for the Sample section;
56	the aggregations carry the totals.
57	
58	## Common pitfalls
59	
60	- Some rules fire every few seconds for healthcheck-style traffic — a
61	  big number is not interesting on its own; look at distribution
62	  across hosts and srcips.
63	- The default Count Breakdown is computed from the post-`--limit`
64	  sample, not the full match set. The CLI labels this when truncated,
65	  but if you need correct totals at scale, use the JSON+aggs form
66	  above.
