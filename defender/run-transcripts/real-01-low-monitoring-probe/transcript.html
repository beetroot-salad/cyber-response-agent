69	### 3. Run the queries
70	
71	**Validate every bound param against the template's declared shape
72	before substitution.** Templates that filter on typed index fields
73	(IPs, timestamps, port numbers, user IDs) call out the expected
74	literal shape and the failure mode. Most SIEM indexes silently
75	return zero matching events when the literal type doesn't match the
76	field type — there is no error to read. Running the query anyway and
77	reporting "0 events" produces a confidently-wrong observation that
78	the defender will weigh as evidence. If a bound param violates the
79	template's declared shape, refuse the dispatch with a
80	"unrunnable: param shape mismatch" summary and stop.
81	
82	Substitute bound params into each template's `## Query` body and
83	execute via the system's CLI (`Bash`). Pass `--run-dir ${run_dir}`
84	and `--position ${position}` — the CLI writes the formatted (or
85	`--raw`) output to `{run_dir}/gather_raw/{position}.json` itself; do
86	not redirect stdout manually. For multi-query dispatches at the same
87	position, suffix `--position` with a single lowercase letter (`0a`,
88	`0b`, `0c`) per `defender/skills/gather/queries/SCHEMA.md`
89	§Multi-query dispatches.
90	
91	**Aggregate at the source, not in your head.** When the lead asks for
92	counts, distributions, or top-N over a population that may exceed the
93	CLI's `--limit` cap, pass a JSON search body to `--query` with an
94	`aggs` clause (OpenSearch terms / date_histogram / cardinality / etc.)
95	rather than a Lucene string. Server-side aggs return true totals over
96	the *full* match set; the default Lucene-string + Count-Breakdown path
97	is a Counter over the limit-capped sample and will silently mislead
98	when match_count > limit. The CLI labels its breakdown clearly when
99	truncated, but the structural fix is: ask for the aggregation you
100	actually need.
101	
102	Also reach for aggs whenever you need an axis the default breakdown
103	does not cover — `syscheck.path`, `data.srcuser` over a 7d span, an
104	hourly histogram, a cardinality estimate. One Bash invocation, one
105	trip to the indexer.

108	
109	For every bullet in `what_to_characterize`, report a value — even if
110	it is "not available" or "not observed." Be specific: exact IPs,

113	#### Smell test before reporting empty / sparse
114	
115	When a query returns no rows or far fewer than the lead expected, do
116	not just report the empty result and stop. Take one round of self-
117	reflection first — these are the smells that catch silent
118	mis-queries:
119	
120	- **Does an empty result make sense for this lead?** A 90-day window
121	  on a populated system showing zero auth events for a known-active
122	  user is suspicious; an alert that just fired naming the entity
123	  guarantees the index has *some* events for it. If the math doesn't
124	  add up, the query is probably wrong.
125	- **Does the unfiltered index have events in this window?** The
126	  Wazuh CLI summary shows "Index event count (unfiltered, same
127	  window)" alongside matching events. If unfiltered is non-zero
128	  and your filter returns zero, your filter is the suspect, not the
129	  data.
130	- **Drop the most specific clause and re-run.** If the broader query
131	  returns events that should have matched the original, one of the
132	  filter values is mis-shaped (wrong field, wrong literal type, NAT
133	  collapse, decoder version drift). Identify which clause was
134	  load-bearing and report the differential.
135	- **Is there a sibling field the data is actually under?** `data.srcip`
136	  vs `data.src_ip`, `data.dstuser` vs `data.user`, `agent.name` vs
137	  `agent.id` — decoder shifts move events between fields. If a query
138	  returns zero on the named field but the unfiltered window is
139	  populated, sample one raw event and check field placement.
140	
141	If after the smell test the empty result is genuine — index is
142	populated, broader query also returns nothing relevant — report
143	"empty (verified: broader-query also empty / unfiltered window
144	populated but filter rules events out / etc.)" so the defender
145	knows which kind of empty it is. Do not run the full debug protocol
146	on your own (that's the defender's explicit dispatch — §Debug
147	leads); one round of smell-check, then report.

177	## Lead kinds
178	
179	Most dispatches are **template leads** — one or more existing or
180	freshly-authored templates from the catalog. Two other lead kinds
181	exist as fallback methodology; the defender names them explicitly in
182	the lead description when they apply.
183	
184	### Composition leads
185	
186	When the lead asks for a correlation across primitives — *was X
187	followed by Y?*, *who was logged in when Z happened?*, *did the auth
188	session spawn unusual processes?* — run the existing primitive
189	templates that measure each side, and **characterize the join in the
190	summary**. Do not mint a "bridge" template that pretends the
191	correlation is itself a primitive measurement.
192	
193	Example: "did anyone modify /etc/passwd on web-03 in the last 24h, and
194	who was logged in then?" → run `wazuh.file-integrity-changes` (filtered
195	to `/etc/passwd`, host `web-03`, 24h window) and `host-query.user-sessions`,
196	then characterize: which mtime, which sessions overlap.
197	
198	### Ad-hoc leads
199	
200	When no template fits and the question is genuinely one-off (not a
201	shape worth memorizing), run the query inline against the system CLI
202	without authoring a template. Capture raw output the same way — the
203	CLI handles `gather_raw/`. In your summary, set the `queries[]` entry
204	`id: ad-hoc` and include the literal query body so the learning loop
205	can still read what ran:
206	
207	```
208	## Queries run
209	- id: ad-hoc
210	  system: wazuh
211	  body: 'rule.id:5503 AND data.dstuser:jsmith AND data.srcip:10.42.7.183'
212	  window: 6h
213	```
214	
215	Use this when authoring would be premature — you only know it's a
216	shape worth memorizing after seeing it twice.
217	
218	### Debug leads
219	
220	When a prior dispatch returned empty and the defender suspects
221	misconfiguration rather than no-events, the defender will dispatch a
222	**debug lead** explicitly. The protocol:
223	
224	1. Confirm the index/host the query targets actually exists (system
225	   health check).
226	2. Broaden the time window by 10× and re-run.
227	3. Drop the most specific filter clause and re-run.
228	4. Drop the next-most-specific clause; iterate until either rows
229	   appear or all filters are stripped.
230	5. Report the differential: "rows appear when `data.srcip` filter is
231	   dropped — likely IP normalization / NAT issue" or "no rows at any
232	   widening — index empty or misrouted."
233	
234	The defender decides what the differential means; you report it.
235	
236	## Discipline
237	
238	- One dispatch in, one summary out. Do not loop, do not propose
239	  follow-ups (debug leads are the defender's explicit dispatch, not
240	  your initiative).
241	- Keep the summary tight — single screen. Push detail to the raw
242	  payload.
243	- Do not echo raw query output back to the defender; that's the whole
244	  point of letting the CLI persist it to `gather_raw/`.
245	- If the lead is genuinely unrunnable (no system, no plausible
246	  template, no entity binding you can construct), say so plainly and
247	  stop. The defender will record the dead end in the investigation
248	  log.

3	---
4	
5	## Goal
6	
7	Retrieve authentication events (sshd login success/failure, sudo, PAM)
8	from the Wazuh alerts index, filtered by any combination of host, user,
9	and source IP. Used to characterize who is logging in where, from where,
10	with what success rate. Covers the by-host and by-user analyst questions
11	under one template — bind whichever filters the lead needs.
12	
13	## What to characterize
14	
15	- Source IP diversity (count of distinct `data.srcip`, top sources)
16	- Username diversity (count of distinct `data.dstuser`, top usernames)
17	  — when not already bound as a filter
18	- Host diversity (count of distinct `agent.name`, top hosts)
19	  — when not already bound as a filter
20	- Auth methods (`publickey`, `password`, etc., counts)
21	- Success/failure ratio (`rule.groups:authentication_success` vs
22	  `authentication_failed`)
23	- Timing pattern (burst, periodic, irregular) over the window
24	- Volume — total events and events/hour
25	
26	## Query
27	
28	```bash
29	python3 soc-agent/scripts/tools/wazuh_cli.py query \
30	  --query 'rule.groups:(authentication_success OR authentication_failed)${host_clause}${user_clause}${srcip_clause}' \
31	  --window ${window} \
32	  --run-dir ${run_dir}
33	```
34	
35	`${host_clause}` is `" AND agent.name:<host>"` when filtering by host,
36	empty otherwise. Same shape for `${user_clause}` (`data.dstuser:<user>`)
37	and `${srcip_clause}` (`data.srcip:<ip>`). Bind whichever the lead
38	requires; leave the rest empty.
39	
40	## Filter binding
41	
42	Filters are mutually composable; bind only the ones the lead actually
43	fixes. Each filter takes a literal value of a specific shape:
44	
45	- `host` → `agent.name:<hostname>` (Wazuh agent name, e.g. `bastion-01.corp`).
46	- `user` → `data.dstuser:<username>` (the destination/target username).
47	- `srcip` → `data.srcip:<IPv4 or IPv6 literal>` (e.g. `10.42.7.183` or
48	  `2001:db8::1`).
49	
50	### REFUSE: hostname bound to `srcip`
51	
52	`data.srcip` is indexed as an IP-typed field. Wazuh silently returns
53	zero events when the literal is not an IP — there is no error, no
54	warning, no type-mismatch signal in the response. The empty result
55	looks identical to "this IP genuinely had no auth activity," and the
56	defender will read it that way.
57	
58	**If the lead's intent is "events where the source is host X":**
59	1. If a separate template resolves X → IP, run that first and rebind
60	   `srcip` to the resolved literal.
61	2. If "host X originated outbound auth" is the actual measurement, this
62	   is **not in scope for this template** — `auth-events` is the
63	   inbound-auth alerts index. Outbound auth is process / sshd-client
64	   telemetry, a different system. Refuse the dispatch and report
65	   "unrunnable: outbound-auth-from-host is not measured by Wazuh
66	   alerts; needs a different system or template."
67	
68	A value bound to `srcip` that is not a parseable IP literal is a
69	configuration error — the gather subagent must refuse to run rather
70	than executing the query and reporting "0 events."
71	
72	## Common pitfalls
73	
74	- NAT collapse: a single `data.srcip` may aggregate many real sources;
75	  inspect username diversity and session ids before claiming "single
76	  origin."
77	- Window edges: same-second bursts straddle window boundaries. Bracket
78	  with a forward lookahead (e.g. `--end T0+60s`) when the alert
79	  timestamp is the leading edge.
80	- Service accounts vs human accounts have very different shapes; cross
81	  with `environment/context/identity-patterns` if available.
82	- Stale credentials cause periodic failures after password rotation —
83	  looks like low-grade brute force but isn't.
84	
85	## Baseline
86	
87	When deviation framing is in play (rate / volume claims, "new source
88	IP" claims), shift the window 7 days earlier (or 30 days for sparse
89	identity patterns) and re-run with the same filter binding. Compare
90	keys per-key — host set, source IP set, timing distribution.

4	---
5	
6	Wazuh is the org's primary SIEM. Authentication events, file integrity
7	monitoring, syscall audit, and rule-correlated alerts all surface here.
8	
9	## CLI
10	
11	The defender and gather dispatch Wazuh queries through the production
12	adapter at `soc-agent/scripts/tools/wazuh_cli.py`:
13	
14	```bash
15	python3 soc-agent/scripts/tools/wazuh_cli.py query \
16	  --query '<Lucene>' \
17	  --window 2h \
18	  --run-dir {run_dir}
19	```
20	
21	Common subcommands:
22	
23	- `query` — search the alerts index. `--query` is polymorphic:
24	  - **Lucene string** (`rule.id:5710 AND agent.name:web-03`) — the CLI
25	    wraps it in a bool with the time-range filter from `--start` /
26	    `--end` / `--window`. Best for "show me events" leads.
27	  - **JSON search body** (`'{"query": {...}, "aggs": {...}}'`) — the
28	    agent owns the entire body, including time filtering. Use this
29	    whenever the lead asks for counts, top-N, or distributions over a
30	    population that may exceed `--limit` — server-side aggs return true
31	    totals over the full match set, while the default Count Breakdown
32	    is a post-`--limit` sample.
33	  Other flags: `--limit` (default 500, max 10000; use `0` for
34	  count+aggs only), `--run-dir` (persists raw payload under
35	  `{run_dir}/gather_raw/` and wraps stdout in salted untrusted-data
36	  delimiters), `--raw` (rarely needed; default already embeds first 3
37	  events' `_source`).
38	- `health-check` — connectivity probe; the defender does not need to
39	  invoke this directly during a run.
40	
41	Lucene is OpenSearch query_string syntax: `rule.groups:sshd AND
42	data.srcip:10.0.0.5`. JSON bodies use the standard OpenSearch search
43	DSL — `query.bool.must` / `query.bool.filter`, `aggs.{name}.terms` /
44	`date_histogram` / `cardinality`, etc.
45	
46	Working query examples + field-level pitfalls live with the templates
47	under `defender/skills/gather/queries/wazuh/`. Reach for those when
48	authoring or grepping for a Wazuh measurement; this SKILL covers only
49	the system surface.
