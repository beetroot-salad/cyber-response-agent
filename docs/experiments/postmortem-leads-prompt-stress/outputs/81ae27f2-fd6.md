+  no-match / near-miss / out-of-window — the lead surfaces the verdict;
+  the hypothesis's `legitimacy_contract` carries the routing

+## Common Pitfalls
+
+- **Treating absence as confirmation.** A miss is *not* a refutation
+  of the alert — it means "not on the approved list," which is the
+  premise the anchor is designed to make explicit. The hypothesis's
+  `legitimacy_contract` decides the disposition; the lead reports the
+  raw verdict.
+- **Confusing classification with sanction.** `environment/context/`
+  files (ip-ranges, identity-patterns) answer *what is it?* — anchors
+  in `environment/operations/` answer *is it allowed?* An IP
+  classified as `internal-monitoring-host` is not, by virtue of that
+  classification, approved to probe a specific target. Always look up
+  the full tuple, not the component. If the tuple's identity does not
+  match the anchor's declared username for that source (e.g., source
+  is the monitoring-host but username is `sensu` while the anchor
+  lists only `nagios`/`zabbix`/`healthcheck`), that is a refutation,
+  not a partial match.
+- **Reading the simulation's source files as evidence.** Anchor
+  corroboration must come from observable operational state (live
+  service status, declared inventory, current connection list). The
+  host-query CLI deliberately blocks `/opt/workloads/` and
+  `/etc/cron.d/` paths — reading those would short-circuit the
+  investigation to the answer key. The grounding is the *combination*
+  of operational signals, not any single one.
+- **Skipping corroboration when the anchor declares it.** If the
+  anchor's `## Confirmation shape` requires an operational signal
+  (cron daemon active, package present, observed cadence consistent),

+  contract is generic, consult each candidate anchor in order and
+  report each verdict separately — they may disagree, and the
+  disagreement is itself diagnostic.
