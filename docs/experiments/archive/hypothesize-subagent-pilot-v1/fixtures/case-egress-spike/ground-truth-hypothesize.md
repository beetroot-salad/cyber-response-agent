The "5.1× over baseline on a known destination" shape means the destination itself isn't the anomaly — the anomaly is *who on v-001 pushed bytes there, and why now*.

All mechanism hypotheses attach upstream of `e-001` (the aggregate egress edge from `v-001` prod-api-01 → `s3-prod-backups` during the window) via `initiated_by` — the proposed parent is the process(es) on v-001 whose sockets accounted for the 2.9 GB, differing by classification.

**Active hypotheses:**

- `?scheduled-backup` (h-001) — proposed parent `{type: process, classification: sanctioned-backup-agent}` (the org's backup orchestration agent or a cron-invoked backup script in the sanctioned inventory). Predicts (p1): process-flow-attribution shows the 2.9 GB contribution originating from a process whose binary path + ancestry match the sanctioned backup agent, AND the backup-system audit log records a backup run for prod-api-01 scheduled or triggered in the 02:00–04:00 window. Refutation shape (r1): contributing process is not a sanctioned backup agent OR no corresponding backup-system audit entry exists.

- `?app-misbehavior` (h-002) — proposed parent `{type: process, classification: application-runtime-bug}` (the Python/Gunicorn app itself producing abnormal egress due to a code/data-path defect — runaway pagination, accidental dataset dump, retry storm writing full payloads). Refutation shape (r1): contributing process is not the app OR the app's code/config in the window matches the last-known-good state with no egress-relevant change.

Refutation shape (r1): contributing process is not a telemetry sidecar OR no prior outage/backlog event correlates with the spike.

**Pitfalls:**

- `?scheduled-backup`: a matching entry in the backup-system audit log is necessary but not sufficient — an attacker who compromises the backup-orchestration plane could trigger an "expected-looking" backup that exfiltrates to an attacker-controlled bucket. Require BOTH sanctioned-agent ancestry AND backup-system entry AND object-storage audit showing only expected-naming-convention keys before grading `++`.
- `?app-misbehavior`: a large legitimate-but-accidental S3 write from the app (e.g., an endpoint that should have paginated dumped the full dataset) is observationally similar to an attacker using an app RCE to stage exfil through the app's own S3 credentials. Do not refute h-004 just because the contributing process is the app; the deploy/config-change correlation and the S3 object-keys tell the rest.
- `?log-shipper-or-telemetry-runaway`: the shipper writing to `s3-prod-backups` (rather than its usual log-aggregator destination) would be structurally unusual — shippers normally write to dedicated aggregator endpoints, not backup buckets. A shipper contributing to s3-prod-backups is already a mild anomaly and warrants treating h-003 with skepticism.
- `?adversary-controlled-exfil`: the S3 bucket resolves to the ORG's own backup bucket (the hostname is `s3-prod-backups.aws-region-a.example`, a legitimate org endpoint). An attacker writing to the org's own bucket is a real pattern (insider threat or compromised credentials with bucket write access) — destination legitimacy does NOT refute h-004. The object-key convention + process attribution carry the refutation weight.
