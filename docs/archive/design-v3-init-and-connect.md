## 1. Problem Statement

The agent requires programmatic access to security systems (SIEMs, EDRs, asset databases) to investigate alerts. Today, connecting a system means hand-writing a CLI script (`wazuh_cli.py`), configuring environment knowledge files, and setting up credentials — a process that requires reading design docs and understanding the internal knowledge structure.

This creates three problems:

- **High barrier to entry** — An analyst who wants to try the agent must understand the adapter pattern, the 4-layer knowledge model, and the file layout before they can run a single investigation.
- **Unscalable tool catalog** — We cannot pre-build and maintain CLI adapters for every security system. There are too many vendors, too many versions, and too many org-specific custom tools. Even covering the popular ones creates a licensing and testing burden.
- **Fragile assumptions** — The current setup assumes direct API access from the execution environment. In practice, most SOC analysts reach their tools through SOAR platforms, identity proxies, VPNs, or bastion hosts.

### What we need

1. A minimal, deterministic init step that creates a working local environment (no LLM).
2. A repeatable `/connect` skill that connects one system at a time — generates the adapter, tests it, scaffolds the knowledge, and commits the result.
3. A `/preflight` script that validates the current state of all connected systems.

### Design principles

1. **Scripts over LLM** — When work can be done deterministically, use a script. The LLM is for decisions that require understanding API documentation, inferring field mappings, and generating code.
2. **Native query languages** — Adapters pass through the system's native query language (SPL, Lucene, KQL). No abstraction layer. The agent already knows these languages from training.
3. **Per-system CLIs** — Each connected system gets its own script (`splunk_cli.py`, `wazuh_cli.py`). No unified `siem_cli.py`. Systems have different capabilities, limitations, and failure modes that a unified interface would mask.
4. **Incremental connection** — The agent is usable after connecting one system. Additional systems are connected over time as needed. There is no "complete setup" gate.
5. **Human review gate** — Every generated adapter is reviewed by the user before it becomes part of the working environment. Generated code does not go live without human approval.

## 2. Execution Model

### Target deployment: analyst workstation

The initial deployment model is a Claude Code plugin running on the analyst's machine. The analyst has Claude Code installed, installs the plugin, and invokes skills interactively.

```
Analyst workstation
├── Claude Code (installed)
├── soc-agent plugin (installed)
├── Local git repo (created by init)
│   ├── knowledge/    — environment-specific configuration
│   ├── scripts/      — adapter CLIs (generated by /connect)
│   └── runs/         — investigation artifacts
└── Network access to security systems (varies)
```

### Local git init

The init step creates a local git repository. This is the working directory for all investigations, knowledge, and generated adapters. The user can push to a remote whenever they choose — to share with teammates, enable CI, or back up their work.

Why local-first:
- **No assumptions about hosting** — Works before the team has decided on GitHub vs GitLab vs internal hosting.
- **Simple to start** — `git init` is one command. Adding a remote is additive.
- **Future-compatible** — Centralized deployment is "clone this repo onto the server." CI is "add a pipeline to the remote." Nothing about the repo structure assumes local execution.

Constraint: **the repo must not assume its own absolute path.** All scripts resolve paths relative to `SOC_AGENT_DIR` or discover them at runtime. This is what makes the repo portable across local, bastion, container, and CI environments.

### Network access reality

In enterprise environments, the path from the analyst's workstation to a security system is rarely direct:

| Access pattern | Description | Prevalence | Agent compatibility |
|---------------|-------------|------------|-------------------|
| VPN + direct API | Analyst VPNs in, has network path to system API | Common | Works directly |
| SOAR/orchestration API | SOAR platform (XSOAR, Phantom, Swimlane) has API access; analyst has SOAR credentials | Common | Adapter calls SOAR API, SOAR calls system |
| Bastion/jump host | Analyst SSHs to secure workstation with system access | Common | Agent runs on the bastion, not the laptop |
| Identity proxy | CyberArk/BeyondTrust vault; just-in-time API keys | Moderate | Credential retrieval is a pre-session step |
| Web UI only | No programmatic access | Varies | Not supported — fundamental blocker |

The adapter layer handles this transparently. Whether `splunk_cli.py` makes a direct REST call or shells out to the org's existing CLI tool doesn't matter to the investigation loop. The `/connect` skill asks how the user accesses the system and generates the adapter accordingly.

### Future deployment models

The local-first model extends naturally:

| Model | What changes | What stays the same |
|-------|-------------|-------------------|
| **Shared server** | Repo is cloned to server, alerts routed via queue | Adapter contract, knowledge structure, investigation flow |
| **Team sharing** | Push to remote, teammates clone | Everything — precedents and knowledge propagate via git |
| **CI/CD validation** | Add pipeline to remote (lint adapters, run preflight) | Repo structure, adapter contract |
| **Container deployment** | Repo mounted or cloned into container image | Adapter scripts, environment knowledge |

No structural changes required. The repo is the unit of deployment.

## 3. Adapter Contract

Every connected system — whether SIEM, EDR, CMDB, or custom tool — exposes its capabilities through an adapter script. The contract is minimal: two required operations, everything else optional.

### Python ABC

```python
"""Adapter contract for security system CLIs.

This ABC defines the interface that /connect generates and that the
investigation loop depends on. Implementations are standalone CLI scripts
(not imported as modules) — this ABC exists for documentation and testing,
not runtime dispatch.

CLI scripts implement this interface through argparse subcommands:
    python3 {system}_cli.py health-check
    python3 {system}_cli.py query <native_query> [--start ...] [--end ...]
"""

from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Any


@dataclass
class HealthResult:
    """Binary health check result.

    connected: did we reach the system and authenticate?
    error:     if not connected, what went wrong (connection refused,
               auth failed, timeout)? Human-readable, one line.
    """
    connected: bool
    error: str | None = None


@dataclass
class QueryResult:
    """Result of a native query execution.

    raw:          the system's response, minimally processed. JSON-serializable.
                  No field renaming, no schema normalization. The agent and
                  post-processing Python handle interpretation.
    total_hits:   how many results matched (before any limit). None if the
                  system doesn't report this.
    truncated:    True if results were cut off by a limit. The agent needs
                  to know when it's seeing a partial picture.
    query_echo:   the exact query that was executed (for audit and debugging).
    """
    raw: list[dict[str, Any]]
    total_hits: int | None = None
    truncated: bool = False
    query_echo: str = ""


class AdapterContract(ABC):
    """Interface that every system adapter implements.

    Two operations. Health check tells you if the system is reachable.
    Query executes a native query and returns results. Everything else —
    field enumeration, aggregation, post-processing — the agent does
    in Python after getting raw results.
    """

    @abstractmethod
    def health_check(self) -> HealthResult:
        """Binary connectivity check.

        Tests: can we reach the endpoint and authenticate?
        Does NOT assess data freshness, index health, or pipeline status.
        Those are unbounded problems — the investigation methodology handles
        them (e.g., checking if zero results means "clean" or "broken").

        Exit codes (CLI):
            0 — connected
            1 — connection or auth failure (print error to stderr)
        """
        ...

    @abstractmethod
    def query(
        self,
        native_query: str,
        time_start: str | None = None,
        time_end: str | None = None,
        limit: int = 500,
    ) -> QueryResult:
        """Execute a query in the system's native language.

        native_query: the query string as the system understands it.
            Splunk adapter receives SPL. Wazuh adapter receives Lucene.
            Elastic adapter receives KQL. No translation layer.

        time_start/time_end: ISO 8601 UTC. Optional — some systems don't
            support time-bounded queries (e.g., asset databases). Adapters
            that don't use time ranges ignore these parameters.

        limit: max results to return. Adapter handles pagination internally
            and sets truncated=True if total_hits exceeds limit.

        Exit codes (CLI):
            0 — query executed (even if zero results)
            1 — query execution failed (print error to stderr)
            2 — connection/auth failure (print error to stderr)
        """
        ...
```

### What the contract deliberately excludes

| Capability | Why excluded | How the agent handles it instead |
|-----------|-------------|--------------------------------|
| Field enumeration | Some systems don't support it; agent can query for it natively or sample and inspect | `query` a wildcard or small sample, then post-process with Python |
| Aggregation | Query language differences make this hard to abstract; agent knows native aggregation syntax | Use native aggregation in the query string, or post-process raw results |
| Result formatting | Formatting is context-dependent (investigation needs vary) | Agent formats results in Python based on what the lead needs |
| Schema normalization | Field names carry semantic information the agent uses (`data.srcip` vs `src_addr`) | No renaming. Agent reads environment knowledge for field mapping context |
| Data freshness | Unbounded problem (what counts as "fresh"?) | Investigation methodology handles this — zero results trigger verification queries |

### CLI interface

Adapters are standalone CLI scripts invoked via Bash. The argparse interface follows a consistent pattern:

```
python3 {system}_cli.py health-check
python3 {system}_cli.py query <native_query> [--start ISO] [--end ISO] [--limit N] [--raw]
```

- `health-check` prints `connected` or `error: <message>` and exits 0 or 1.
- `query` prints formatted results to stdout. `--raw` outputs JSON.
- `--run-dir` (optional) reads the per-run salt from `meta.json` and wraps output in untrusted-data delimiters.

### Credential handling

Adapters are the trust boundary for credentials. The agent never handles raw secrets.

```
Environment variables (secrets)     Config file (non-secrets)
  SPLUNK_TOKEN                        config.env: endpoint, index, defaults
  WAZUH_INDEXER_PASSWORD              ↓
        ↓                           Loaded by adapter at startup
  Injected into adapter process      ↓
        ↓                           Agent sees: query results, health status
  Agent constructs query string     Agent never sees: tokens, passwords, endpoints
  Agent calls: python3 splunk_cli.py query "index=main ..."
```

Credentials are set up outside the agent's scope — via `.env` file, shell export, vault integration, or however the org manages secrets. The `/connect` skill tells the user which environment variables to set but never asks for the values.

### MCP coexistence

Some systems may already be accessible via MCP servers. The adapter contract applies regardless of the underlying mechanism:

| Situation | What /connect generates |
|-----------|----------------------|
| No MCP server, API docs available | CLI script that implements the contract directly (HTTP calls) |
| No MCP server, existing org CLI | CLI script that wraps the existing tool and parses its output |
| MCP server with few tools (<5), frequently used | Recommendation to use MCP directly; generates environment knowledge docs but no CLI. Lead templates reference MCP tool names |
| MCP server with many tools (>10), rarely used | CLI script that calls specific MCP tools; avoids loading full schema into context |

The decision is per-system: **how much context does the MCP server consume vs how often is it used?** The `/connect` skill surfaces this tradeoff to the user and recommends, but the user decides.

## 4. Init Flow

### 4.1 Overview

```
1. Install plugin (manual)
   └── claude plugin add ./soc-agent

2. Init (script — deterministic, no LLM)
   └── scripts/init.sh
       ├── git init (if not already a repo)
       ├── Create directory structure
       ├── Install Python deps (core — stdlib only)
       └── Print next steps

3. /connect (LLM skill — interactive, repeatable)
   ├── First invocation: connect primary event query system
   └── Subsequent invocations: connect additional systems
       ├── Interview: what system, what type, how accessed
       ├── Generate adapter CLI
       ├── Test: health check + sample query
       ├── Scaffold environment knowledge
       └── Commit (branch + PR-style review)

4. /preflight (script — idempotent, run anytime)
   ├── Run all adapter health checks
   ├── Validate knowledge base structure
   └── Report: ready / degraded / not configured
```

### 4.2 Init script (`scripts/init.sh`)

Deterministic setup. No LLM involvement. Runs once after plugin installation.

**Preconditions:**
- Python 3.11+
- Git
- Claude Code installed with API key configured

**What it does:**

```bash
# 1. Git init (skip if already a repo)
#    Creates .gitignore (runs/, .env, __pycache__, .venv/)

# 2. Directory scaffold
#    knowledge/environment/context/       — org context (user-written)
#    knowledge/environment/sources/       — data source docs (generated by /connect)
#    knowledge/environment/systems/       — per-system config (generated by /connect)
#    knowledge/signatures/                — signature knowledge (user-written)
#    scripts/tools/                       — adapter CLIs (generated by /connect)
#    runs/                                — investigation artifacts (ephemeral)

# 3. Validate Python version

# 4. Print next steps
#    "Run /connect to connect your first security system."
#    "Run /preflight to check your setup."
```

**What it does NOT do:**
- Install SIEM-specific dependencies (that's per-adapter, handled by `/connect`)
- Configure credentials (that's the user's responsibility)
- Create or modify Claude Code settings (that's the plugin manifest's job)
- Touch anything outside the repo directory

### 4.3 The `/connect` skill

An interactive, LLM-driven skill that connects one system per invocation. The user runs it when they want to add a new data source — on day one for their primary SIEM, months later when they want EDR coverage.

**Invocation:**

```
/connect                          # interactive — skill asks what to connect
/connect splunk                   # shorthand — skill knows the target system
```

#### Phase 1: Interview

The skill asks targeted questions to understand the access path:

```
1. What system are you connecting?
   (e.g., Splunk, Elastic/OpenSearch, CrowdStrike, custom REST API)

2. How do you access it from this machine?
   a) Direct API (I have an endpoint and credentials)
   b) Through a SOAR/orchestration platform
   c) Via an existing CLI tool or script
   d) Through an MCP server
   e) Other (describe)

3. [If API] What's the API endpoint?
   [If SOAR] What SOAR platform and what command/integration?
   [If CLI] Where is the script and how do you invoke it?
   [If MCP] What's the server name and what tools does it expose?

4. What data does this system hold that's relevant to investigations?
   (e.g., authentication events, process execution, network flows,
    asset inventory, identity/role data)
```

The skill does NOT ask for credentials. It tells the user which environment variables to set and waits for confirmation that they're configured.

#### Phase 2: Generate adapter

Based on the interview, the skill generates a CLI script that implements the adapter contract:

| Access path | What gets generated |
|------------|-------------------|
| Direct API | `scripts/tools/{system}_cli.py` — HTTP client using stdlib (`urllib`) or a minimal dependency. Includes `health-check` and `query` subcommands. Config loaded from `knowledge/environment/systems/{system}/config.env`. |
| SOAR/orchestration | `scripts/tools/{system}_cli.py` — wraps SOAR API calls. The adapter calls the SOAR, which calls the target system. Credentials are for the SOAR, not the target system. |
| Existing CLI | `scripts/tools/{system}_cli.py` — thin wrapper that invokes the existing tool via subprocess, parses its output, and returns it in the contract format. |
| MCP server (chosen as primary) | No CLI generated. Skill creates environment knowledge docs and notes MCP tool names. Lead templates reference MCP tools directly. |

The generated script follows the patterns established by `wazuh_cli.py`:
- Config from `config.env` (non-secrets), env vars for secrets
- Clear error messages pointing to setup steps when deps or config are missing
- Graceful import failure (print setup command, exit 2)
- `--run-dir` support for salt-wrapped output
- `--raw` for JSON output

If the adapter needs external dependencies (e.g., `opensearch-py`, `splunk-sdk`), the skill adds a named extra to `soc-agent/pyproject.toml` and updates `uv.lock` via `uv sync --extra dev`. See [packaging.md](packaging.md).

#### Phase 3: Test

The skill runs the generated adapter to verify it works:

```
Step 1: Health check
  $ python3 scripts/tools/splunk_cli.py health-check
  → connected / error: <message>

Step 2: Sample query (if health check passes)
  $ python3 scripts/tools/splunk_cli.py query "<simple_query>" --limit 5
  → Shows sample results to user for validation
  → "Do these results look right?"
```

If the health check fails, the skill diagnoses the error (connection refused → network issue, 401 → credential issue, timeout → endpoint issue) and guides the user through fixing it.

If the sample query returns unexpected results, the user can iterate with the skill to fix the adapter before committing.

#### Phase 4: Scaffold knowledge

For each connected system, the skill creates environment knowledge docs:

```
knowledge/environment/
├── sources/
│   └── {system}-{instance}.md      # What data lives here, coverage notes
└── systems/
    └── {system}/
        ├── config.env              # Non-secret configuration (endpoint, index, etc.)
        ├── config.env.template     # Template for sharing (secrets redacted)
        └── field-notes.md          # Initial field documentation (populated over time)
```

The `sources/` doc maps data coverage:

```markdown
# Splunk (Main Instance)

## Coverage
- Authentication events (Windows Event Log, Sysmon)
- Process execution (Sysmon)
- Network connections (firewall logs)

## Access
- Adapter: `scripts/tools/splunk_cli.py`
- Query language: SPL
- Retention: 90 days (per user input)

## Notes
- [Generated by /connect on 2026-04-06]
- Field documentation in systems/splunk/field-notes.md
```

#### Phase 5: Commit

The skill creates a branch, commits the generated files, and presents a summary:

```
Created:
  scripts/tools/splunk_cli.py          — adapter (health-check + query)
  pyproject.toml                       — [splunk] extra added (splunk-sdk>=x.y)
  uv.lock                              — updated
  knowledge/environment/systems/splunk/config.env
  knowledge/environment/systems/splunk/config.env.template
  knowledge/environment/systems/splunk/field-notes.md
  knowledge/environment/sources/splunk-main.md

Health check: connected
Sample query: 5 events returned (authentication events from last hour)

Review the generated adapter and merge when ready.
```

The user reviews the diff, potentially edits the adapter, and merges. The agent does not merge — the human review gate is the trust boundary for generated code.

### 4.4 Preflight script

A deterministic check that validates the current environment. No LLM — just a Python script that runs health checks and validates file structure.

```
$ python3 scripts/preflight.py

Systems:
  ✓ wazuh     — connected (indexer: 127453 docs, manager: 42 agents)
  ✓ splunk    — connected
  ✗ crowdstrike — error: connection refused (check VPN?)

Knowledge base:
  ✓ signatures/wazuh-rule-5710 — complete (context, playbook, 3 precedents)
  ! signatures/wazuh-rule-5503 — missing playbook.md

Environment:
  ✓ context/ip-ranges.md — present
  ! context/identity-patterns.md — empty (template only)

Result: DEGRADED (1 system unreachable, 1 incomplete signature)
```

Exit codes:
- 0 — all systems connected, knowledge base complete
- 1 — degraded (some systems unreachable or knowledge gaps)
- 2 — not configured (no systems connected)

Preflight is idempotent. Run it after `/connect`, after credential rotation, after network changes, or as a periodic health check. It surfaces problems before they appear mid-investigation.

## 5. How This Integrates with Investigation

The investigation loop is unchanged. The difference is in how tools become available:

### Before (current state)

```
Hand-write wazuh_cli.py → manually create config.env → manually create
environment knowledge docs → manually wire into lead templates → investigate
```

### After

```
/connect wazuh → adapter generated + tested → knowledge scaffolded →
committed → /preflight validates → /investigate uses it
```

### Tool resolution during investigation

When a lead needs data:

```
playbook.md: "pursue authentication-history lead"
    ↓
lead definition.md: data_tags = [auth-events]
    ↓
knowledge/environment/sources/: "auth events are in Splunk and Wazuh"
    ↓
lead templates/splunk.md: field mapping + base query in SPL
    ↓
scripts/tools/splunk_cli.py query "index=main sourcetype=WinEventLog ..." --start ... --end ...
    ↓
Raw results → agent post-processes in Python
```

If a lead needs data from a system that isn't connected, the fail-fast pattern surfaces it:

```
lead definition.md: data_tags = [edr-events]
    ↓
knowledge/environment/sources/: no source covers edr-events
    ↓
Agent reports: "This lead requires EDR data. No EDR system is connected.
Run /connect to add one, or skip this lead and note the coverage gap."
```

This creates natural pressure to connect additional systems without making it a gate.

### Lead templates for connected systems

When `/connect` generates an adapter, it does NOT auto-generate lead templates. Lead templates encode investigative methodology (what fields matter, what to characterize, common pitfalls) — that's knowledge that comes from investigation experience, not from API documentation.

What `/connect` does provide is the foundation: the adapter script, the field documentation, and the data coverage mapping. Lead templates are authored over time as the team investigates alerts and learns what queries are effective.

The exception: if the connected system is well-known (Splunk, Elastic, CrowdStrike), the skill may generate starter templates for common leads (authentication-history, process-execution) based on the agent's training knowledge. These are clearly marked as drafts for human review.

## 7. Open Questions

### Adapter testing

Generated adapters should have tests, but what kind? Options:
- **Contract compliance test** — Generic test that any adapter can be plugged into: does `health-check` return the right exit code? Does `query` return valid JSON with `--raw`? This can be a single test parameterized over all adapters in `scripts/tools/`.
- **Integration test per adapter** — Specific to each system (requires live access). Useful but hard to run in CI without credentials.

Likely answer: contract compliance tests are generated by `/connect`, integration tests are manual.

### Config file format

Currently `config.env` (shell-sourceable key=value). This works but doesn't support structured data (lists, nesting). If adapter config grows beyond simple key-value pairs, consider TOML or stay with env-style and use JSON for complex values.

Likely answer: stay with `config.env` for now. Adapters that need structured config can use a separate JSON file alongside it.

### Adapter updates

When a system's API changes or the user wants to modify the adapter, they can either edit the script directly or re-run `/connect` for that system. Re-running `/connect` should detect the existing adapter and offer to update rather than overwrite.

### Non-query systems

Some useful systems don't expose a query interface — e.g., a CMDB with a lookup-by-hostname API, or a threat intel platform with an IoC enrichment endpoint. These don't fit the `query(native_query)` contract cleanly. Options:
- Separate contract for lookup/enrichment systems (different ABC with `lookup(identifier)` instead of `query`)
- Ad-hoc scripts that don't implement the contract (agent calls them directly)
- Stretch the query contract (the "native query" is the lookup key)

Likely answer: ad-hoc scripts for now. If a pattern emerges across multiple lookup-style integrations, formalize it then.
