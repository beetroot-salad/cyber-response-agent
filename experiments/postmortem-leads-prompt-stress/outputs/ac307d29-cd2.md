+name: actor-attribution
+data_tags: [process-events, asset-state]
+baseline: not-applicable   # The lead answers a binary attribution question — is the expected scheduled process on the source host alive AND did matching activity register in the SIEM at t-0 ±jitter. Cadence/volume claims belong in `authentication-history` or `network-analysis`, which carry their own shift-baseline machinery.
+---
+
+## Goal
+
+Attribute a network or authentication event to a specific actor on
+the source host by combining live host-state on the source with a
+narrow SIEM time-correlation around the alert timestamp.
+
+This lead applies when the alerting event's `srcip` resolves to an
+internal host that is *expected* to be running a known scheduled
+process (a monitoring agent, a scheduled job runner, a known service
+account's cron-driven client) and the question is whether *that*
+process produced the observed action — as distinct from a different
+actor on the same host or a spoofed source.
+
+## What to Characterize (required output)
+
+Each bullet below MUST be reported on, even if the answer is "not
+available in this data source" or "not observed."
+
+- **Expected actor liveness**: Is the named scheduled process /
+  service currently active on the source host? Use the live
+  host-state query (process name, service status). Report
+  `alive | inactive | absent`. Name the specific process / service
+  checked.
+- **Temporal alignment**: SIEM events on the source IP within ±2 min
+  of the alert. Report event count, distinct rule.ids, and the
+  closest event's offset from the alert timestamp. An empty window
+  is itself a finding when paired with a "process is alive"
+  finding from the host-side query.
+- **Discriminator fields**: For each correlated SIEM event, the
+  fields that would distinguish the expected actor's traffic from a
+  sibling process on the same host (source port pattern, command-line
+  fragment if visible, target tuple, identity-of-use field). Pull
+  raw events to see these — the summary projection collapses them.
+- **Coverage gap**: When the source host has no endpoint agent or the
+  expected scheduling channel has no audit trail, say so explicitly.
+  Live state at query time does not prove the same process was the
+  actor at alert time, and a missing audit trail caps the certainty
+  the lead can deliver.
+
+## Common Pitfalls
+
+- **Liveness ≠ identity-of-use.** A monitoring tool is alive and the
+  source/username triple is registered, but the alerting event could
+  be from a different actor on the same host — an operator running a
+  sibling script, a compromised process reusing the credential
+  string, or a tunnel hijack. Liveness is necessary but not
+  sufficient; pair it with the temporal alignment and the
+  discriminator-field check.
+- **Live-now is not live-then.** `host_query process-list` (or the
+  equivalent live-state adapter) returns the process table at query
+  time. If the alert fired hours ago, a process running now may not
+  be the process that ran then. Where past-process telemetry is
+  available on the source, prefer it; this lead is the fallback
+  when it is not.
+- **Wide correlation windows admit coincidence.** ±2 min is the
+  default — tight enough to attribute the action when one alert is
+  in scope. Widening to hours pulls in unrelated traffic and erodes
+  the attribution.
+- **Process-name impersonation.** A binary may be reported under a
+  name that doesn't match its actual executable. When the live-state
+  query returns only names (no argv, no exepath), confirmation is
+  weaker than when the adapter exposes the executable path.
+
+## Templates
+
+Per-vendor query templates live under `templates/{vendor}.md` and
+cover the SIEM-correlation half. The live host-state half is
+deployment-specific and lives in
+`environment/systems/{system}/SKILL.md` (e.g.,
+`environment/systems/host-query/SKILL.md` for the playground).
+
+## Cross-references
+
+- `process-lineage` reconstructs the process tree on a host with
+  endpoint telemetry, working backward from a suspicious child. It
+  assumes past-process events are available on the host that ran
+  the process. `actor-attribution` is its counterpart for the source
+  side, where past-process telemetry is typically absent and the
+  question is identity-of-use of the *expected* scheduled actor
+  rather than ancestry of an unknown one.
+- `authentication-history` and `network-analysis` answer cadence,
+  volume, and rate claims with shift-baseline comparisons. Use them
+  when the question is "does the observed shape match the historical
+  baseline for this tool's traffic," not "is the expected actor the
+  one producing this event."
+- `approved-monitoring-sources` answers registration (is the triple
+  *allowed*?). It is orthogonal to this lead, which answers
+  identity-of-use (is the registered actor the one producing
+  *this* event?).
